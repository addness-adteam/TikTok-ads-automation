# 日次運用OS - 思考ルール

このファイルは運用OSのTodo生成に適用される「考え方」を蓄積する場所です。
ユーザーのFBを受けてルールを追加・更新し、Todoの精度を継続的に向上させます。

## ルールの書き方

各ルールは `rule` コードブロックにJSON形式で記述します。
スクリプト（daily-ops-todo.ts）が起動時に読み込み、各Todoに自動適用します。

**フィールド:**
- `id`: ルール識別子（ユニーク）
- `category`: 適用カテゴリ（"再出稿", "横展開", "Smart+", "LP", "停止CR", "時間帯", "個別予約CPO" のいずれか）
- `rule`: ルールの説明（人間が読む用）
- `condition`: 適用条件（`metricsキー 演算子 値` 形式、省略可）
- `action`: Todoに追加される補足アクション

## 現在のルール

### R001: 前日個別予約CPO → 再出稿判断
```rule
{"id":"R001","category":"個別予約CPO","rule":"直近3日で個別予約が入ったCRはCPOを算出し、KPI以内なら再出稿候補として表示。予約0件のCRは別途確認","action":"個別予約KPI以内→予算増or再出稿を優先検討"}
```

### R002: 個別予約0件でCV多い → ピクセル/LP確認
```rule
{"id":"R002","category":"再出稿","rule":"CVは出ているが個別予約0のCRはピクセル or LP側の問題がないか確認","condition":"cv > 10","action":"個別予約0件の場合、ピクセル発火/LPフォーム遷移を要確認"}
```

### R003: 枯れCR → 動画差し替え or 別アカウント
```rule
{"id":"R003","category":"再出稿","rule":"同一CRが直近で複数回停止されている場合は枯れの可能性。動画フック差し替えか別アカウントでの出稿を検討","condition":"isStale == 1","action":"枯れ検知→フック差し替え or 別アカウント出稿を優先"}
```

### R004: 横展開は実績ベース（スプシの数字で判断）
```rule
{"id":"R004","category":"横展開","rule":"横展開候補はスプシの実績（個別予約CPO）で判断。感覚ではなく数字ベース","action":"必ずスプシの個別予約CPOを確認してから横展開実行"}
```

### R005: CV出てもフロント売れないCRは再出稿しない
```rule
{"id":"R005","category":"再出稿","rule":"CVは出るがフロント商品が売れない（フロントCPOが合わない）CRは、ターゲティング変更・アカウント変更しても改善しない傾向。再出稿候補から除外。例: CR00604はSNS3初出稿→CV出るがフロント売れず→25-44ターゲで再出稿→CV伸びず→SNS2横展開→同結果。訴求自体がフロント購入に繋がらないと判断","condition":"indRes == 0","action":"フロント販売0のCRは訴求そのものの問題。ターゲ/アカウント変更では改善しない可能性大"}
```

### R006: 既に再出稿済みのCRは再出稿候補から除外→効果測定に回す
```rule
{"id":"R006","category":"再出稿","rule":"すでに再出稿・横展開済みのCRを再度再出稿候補として表示しない。代わりに効果測定（現状確認）として表示","action":"再出稿済み→効果測定フェーズ。現状の数値を確認してから次の判断へ"}
```

### R007: Smart+出稿直後は効果測定優先
```rule
{"id":"R007","category":"Smart+","rule":"Smart+を新規出稿した直後は、含まれるCRの重複確認＋効果測定を優先。良ければ横展開/複製、悪ければ原因言語化して次に繋げる","action":"Smart+出稿後→まず効果測定。結果が出てから横展開/複製の判断"}
```

### R008: LP-CR番号だけでなく広告名（制作者/CR名）も必ず表示
```rule
{"id":"R008","category":"個別予約CPO","rule":"LP-CR番号だけでは動画が特定できない。広告名（制作者/クリエイティブ名）を併記して、どの動画か即座に判断できるようにする","action":"広告名を見て出稿/横展開の判断をする"}
```

### R009: CV/フロント販売数はスプシから取る（アカウント単位も）
```rule
{"id":"R009","category":"全般","rule":"CV・フロント販売数・個別予約数は必ずスプレッドシート（UTAGE経由）から取得する。TikTok APIのconversionだけに頼らない。アカウント単位のCVも、そのアカウントに存在する広告名→LP-CRを特定→スプシの登録経路からCV/フロント/個別予約を集計する。予算調整V2と同じロジック。理由: TikTokのピクセルが正しく発火していなくてもUTAGE側では登録が入っている場合がある。SNS2のようにTikTok API上CV=0でもスプシ上ではオプトが入っている可能性を見逃さないため","action":"効果測定時はDBメトリクス(TikTok API)とスプシ(UTAGE)の両方を突合。差異があれば警告表示"}
```

### R010: 累計個別予約CPOで再出稿/停止判断
```rule
{"id":"R010","category":"再出稿","rule":"再出稿判断や停止判断の最終根拠は累計個別予約CPO。CPAが良くても個別予約CPOがKPI超過なら停止検討。例: CR00223はCPA¥2,751で好調だが個別予約CPO¥242,103でKPI4.5倍超過→停止判断","action":"CPA好調でも個別予約CPOがKPI超過なら停止を提案"}
```

### R011: スキルプラス導線ではSmart+を使わない（CPC問題）
```rule
{"id":"R011","category":"Smart+","rule":"スキルプラス導線でSmart+配信するとCPCが¥10〜30に急低下し、質の低いクリックが増加。オプトイン率・リストイン率が低下し、リストインCPAが通常配信の2.2倍に悪化→個別予約CPOも悪化する。スキルプラスではSmart+を使わず通常配信を基本とする。もしSmart+を使う場合はCPC¥100を下回っていないか常に監視し、¥100未満なら停止検討","action":"スキルプラスのSmart+化提案は除外。既存Smart+はCPC¥100未満なら停止警告"}
```
