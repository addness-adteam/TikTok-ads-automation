# AI駆動開発フロー（並列レビュー＆E2Eテスト）

## 概要

AIにコードを書かせた後、複数観点のレビューをAIに並列実行させ、品質を担保してからPRを出す開発フロー。
Webアプリ開発やプロダクト開発で特に有効。スクリプト開発にも応用可能。

## 全体フロー

```
計画（人間が承認）
  ↓
設計 → テスト実装 → プロダクト実装 → 静的チェック → 自動テスト実行
  ↓
ローカルレビュー（4並列 × 最低3回ループ）
  ↓
E2Eブラウザテスト
  ↓
PR作成 → マージ
```

## レビューフェーズの詳細

### 4つのレビュー観点（サブエージェントで並列実行）

| 観点 | チェック内容 |
|---|---|
| **アプリ** | ビジネスロジックの正しさ、仕様との整合性、エッジケース |
| **セキュリティ** | インジェクション、認証/認可、データ漏洩、OWASP Top 10 |
| **QA** | テストカバレッジ、境界値、異常系、デグレチェック |
| **アーキテクチャ** | 全体構成との整合性、設計思想の違反、構造的問題 |

### ループルール

- **3回連続で指摘ゼロ**になるまで繰り返す
- 指摘が出たら修正→回数リセットして最初から3回
- 軽微すぎる指摘は対応不要と判断してよいが、理由を記録する

### レビュー精度を上げるコツ

- **書いたAIと別のモデルでレビューする**（同じ盲点にハマらない）
  - 例: Claude Opusで実装 → Codexでレビュー、またはその逆
- レビュー観点ごとに専用のスキル/プロンプトを用意する

## E2Eテスト（Webアプリの場合）

### 目的
ユーザーが実際にやる操作を自動で再現し、最初から最後まで通しで動くか確認する。

### ツール例
- Playwright
- Cypress
- Selenium

### フロー
1. ブラウザを自動操作してユーザーシナリオを再現
2. 問題が出たら → 静的チェックからやり直し（レビュー回数もリセット）
3. 問題なし → PR作成へ

## スクリプト開発への応用

Webアプリでない場合も同じ考え方を適用できる:

| Webアプリ | スクリプト |
|---|---|
| ブラウザE2Eテスト | スクリプト実行→出力結果の自動検証 |
| アプリレビュー | ビジネスロジックレビュー |
| セキュリティレビュー | APIキー漏洩・データ保全チェック |
| QAレビュー | エッジケース・エラーハンドリング |
| アーキテクチャレビュー | スクリプト間の依存・重複チェック |

## PR作成時の報告内容

PRには以下を含める:
- 計画時と実装時の差分の詳細
- レビュー指摘で対応したもの＆対応方法
- 対応不要と判断したもの＆その理由

## 要点

- **人間は意思決定（計画承認・最終マージ）に集中**
- **品質はレビューの物量（並列×ループ）で担保**
- **レビュアーは実装者と別のAIモデルが望ましい**
- **E2Eテストが最終関門。失敗したら全部やり直し**
